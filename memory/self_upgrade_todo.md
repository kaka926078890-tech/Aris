# 自升级模块 · 当前方案与能力（待进一步思考）

> 记录自升级模块的现状与能力，便于后续想透彻后迭代。实现见 `src/dialogue/selfUpgrade.js`，设计依据见 `memory/self_upgrade_and_lowpower_solution.md`。

---

## 一、当前方案概要

- **触发时机**：由 proactive 在「即将进入低功耗」时调用。即：`proactive_no_reply_count === 2` 且本周期会发出一条 proactive 消息，且 `today_off_work && !self_upgrade_done_today` 时，先执行一次自升级，再标记今日已自升级并进入低功耗。每日最多 1 次。
- **输入**：近期对话（当前会话最近 12 条）+ 行为规则片段（`src/dialogue/rules.md` 前 2000 字）。
- **流程**：构建自升级专用 prompt → 单次 LLM 调用 → 解析回复中的「理由 / 目标 / 修改的内容 / 结论」→ 若解析到可选的 `FILE:` + `CONTENT:` 且路径在白名单内则执行一次 `writeFile` → **无论是否改文件，都**将当次反思完整追加到 `memory/self_upgrade_log.md`。
- **白名单**：仅允许修改 `src/dialogue/*.js`、`memory/*.md`（相对项目根）；其他路径一律不执行写入。

## 二、当前能力

| 能力 | 说明 |
|------|------|
| 反思与记录 | 用近期对话 + 规则片段让 LLM 输出「理由、目标、修改的内容、结论」，并持久化到 `self_upgrade_log.md`，便于审阅与回溯。 |
| 结构化解析 | 从 LLM 回复中解析 `### 理由`、`### 目标`、`### 修改的内容`、`### 结论` 四个区块。 |
| 可选自动改文件 | 若「修改的内容」里包含 `FILE: 相对路径` + `CONTENT:` + 完整文件内容，且路径在白名单内，则执行一次覆盖写入；否则只记录文字说明，不执行写入。 |
| 安全边界 | 路径校验：禁止 `..`、仅允许上述前缀与 `.js`/`.md` 扩展名；非白名单路径只打 log，不写入。 |

---

## 三、当前限制与待思考点

- **自动改文件的可靠性**：依赖 LLM 一次性输出完整文件内容，易出现格式错误、漏写、截断；是否改为「只写日志、改代码由人工审阅后再做」或增加「先 read 再 diff 再写」的流程，尚未定。
- **上下文范围**：目前仅用最近 12 条对话 + rules 前 2000 字，未接入更多代码片段、记忆或 self_upgrade_log 历史；后续是否要扩展上下文（如注入相关源文件、历史升级记录）待定。
- **单轮调用**：一次 LLM 调用即结束，没有「先 read_file 再根据内容写回」的工具链；若希望模型基于当前文件内容做增量修改，需要改为支持多轮工具调用或在本轮 prompt 中明确要求「给出完整新内容」。
- **与低功耗的衔接**：自升级执行完毕后会标记 `self_upgrade_done_today` 并进入低功耗，不再发 proactive 消息；若自升级过程中抛错，目前仍会进入低功耗（可考虑：失败时是否不计入「今日已自升级」、是否重试等）。
