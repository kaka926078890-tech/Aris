# 提示词注入说明：必定注入 vs 条件注入

本文档说明每轮 system prompt 的注入内容、篇幅约定，以及常见问题（用户要求提炼、行为规则等）。

---

## 一、必定注入（每轮都会进入 system）

| 注入项 | 数据来源 | 内容量/上限 | 说明 |
|--------|----------|-------------|------|
| **人设 + 对话规则** | `persona.md` + `conversation_rules.md`（或默认） | 约 800～1500 字 | 人格、回复要求、【用户约束】说明等 |
| **【用户身份】** | `facade.getIdentity()` | 1～3 行 | 姓名 + 备注，无则「（无）」 |
| **【用户约束】** | 见下 | 见下 | **统一块**：用户要求 + 纠错记录 + 用户喜好 + 禁止用语 |
| **【你上一次的状态与时间感】** | `facade.getState()` | 1～2 行 | 当前时间 + 距上次活跃 + 上次状态/想法 |
| **【相关关联】** | `getRelatedAssociationsLines()` | 1～10 行（默认 3） | 身份/要求关联，无则「（无）」 |
| **【近期小结】** | `facade.getSessionSummary(sessionId)` | 一段 | 当前会话最新小结，无则「（无）」 |
| **【当前会话最近几轮】** | `recent` | **6 条**（最近 3 轮） | 用户/Aris 交替，带时间 |
| **【行为规则】** | `rules.md` | 视文件 | 全局行为规则，无则「（无）」 |
| **行为边界 + get_my_context 提示** | 配置 + 写死 | 各 1 句 | 读代码边界、调用 get_my_context |
| **fetch_url 提示** | 仅当网络开启 | 1 句 | 可选 |

### 【用户约束】块（每轮完整注入）

以下四类**每轮都会**拼成一块注入，不再「按需调工具」：

| 子块 | 数据来源 | 量级 |
|------|----------|------|
| **用户要求** | `facade.getRequirementsSummary()` | 完整汇总，无条数上限 |
| **纠错记录** | `facade.getCorrectionsFullSummary()` | 完整汇总，无条数上限 |
| **用户喜好** | `facade.getPreferencesSummaryForPrompt()` | 完整汇总，无条数上限 |
| **禁止用语** | `facade.getAvoidPhrasesForPrompt()` | 一行「禁止使用以下表述：xxx、yyy」或未配置时不注入 |

目的：要求、纠错、喜好、禁止统一成一块，模型每轮都能看到完整约束，无需再调 get_record 等工具才遵守。

---

## 二、条件/动态注入（满足条件才追加）

| 注入项 | 触发条件 | 量级 |
|--------|----------|------|
| **最近情感一句** | 行为配置 `inject_recent_emotion` 开启 且 有情感记录 | ≤40 字 |
| **fetch_url 提示** | 网络开关开启 | 1 句 |

## 三、常见问题（FAQ）

### 1. 用户要求 / 纠错 / 喜好新增时会不会做总结？

**会。** 在 `store/requirements.js`、`store/corrections.js`、`store/preferences.js` 中，**每次新增**后都会在后台自动调用文档式总结（`refineToDocument`）：将原先内容 + 新内容合并为一份完整文档并写回，不遗漏信息。管理页点击「保存」时也会先写入再触发总结并刷新。

### 2. 行为规则是怎么触发更新的？有没有地方可以直接编辑？

- **来源**：`rules.md` 从**项目目录**读取，路径为 `v2/rules.md`（相对于 `packages/server/dialogue/prompt.js` 的 `RULES_PATH`）。
- **更新方式**：没有「触发更新」的机制，也没有应用内编辑入口。改内容 = 直接编辑 **`v2/rules.md`** 文件，保存后下一轮对话会读到新内容（每次 build 会重新 `loadRules()`）。
- 若希望可在应用里改，需要后续在「设置/管理」里增加对 `rules.md` 的读写（与 conversation_rules、persona 类似）。

### 3. 纠错、喜好、禁止用语为何要统一注入？

- **纠错**：每轮在【用户约束】里注入 **纠错完整摘要**（全部内容，总结为一份文档），保证纠错记录完整、可被遵守。
- **用户喜好**：每轮在【用户约束】里注入 **用户喜好摘要**（全部内容，总结为一份文档），不再依赖「按需 get_record(preferences)」。
- **禁止用语**：每轮在【用户约束】里注入 **禁止用语列表**（来自 `memory/avoid_phrases.json`）；未配置时不注入该块。
- **统一**：要求、纠错、喜好、禁止 四类都作为「新内容 + 旧内容」的完整块，每轮统一放在【用户约束】中，模型按同一块遵守即可。

## 四、篇幅与上限汇总

| 类型 | 项 | 说明 |
|------|----|------|
| 固定 | 人设 + 对话规则 | 约 800～1500 字 |
| 固定 | 行为边界 + get_my_context + 可选 fetch_url | 约 80～120 字 |
| 可变 | 用户身份、**【用户约束】**、状态与时间、相关关联、近期小结、当前会话几轮、行为规则 | 无硬编码条数上限，完整注入 |
| 条件 | 最近情感一句、fetch_url 提示 | 见上表 |
