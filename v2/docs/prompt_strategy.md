# v2 提示词策略

## 采用方案 A（后续可换方案 B）

- **首轮**：发送完整 system prompt（人设、行为规则与约束、工具定义、当前用户身份与要求摘要、上次状态与时间感等）。
- **后续轮**：采用 **方案 A**：
  - 每轮仍发送 **人设 + 核心行为规则/约束**（可压缩为一段，但必须包含）。
  - 每轮仍发送 **完整工具定义**（通过 API 的 tools 参数），保证模型了解能力并会调用工具。
  - **可精简部分**：用户身份、用户要求、长段记忆等，在后续轮中改为 **固定注入简短摘要**（如 1～2 句）或从 store 读出的最近条目摘要，不再每轮重复整段长文本；对话历史照常发送。

若后续希望减少 token 或更依赖按需拉取，可切换为 **方案 B**（后续轮极简 system + 身份/要求/记忆一律通过工具按需获取）。当前实现与文档以方案 A 为准，方案 B 作为可选替代，切换时需同步更新本说明与 `server/dialogue/prompt.js`。

## 为何每轮都要带人设与工具

API 按请求无状态：每轮只看到当前请求里的 system + messages。首轮发过的内容在后续轮不会自动保留，因此每轮都需包含人设、约束与工具定义，模型才能持续遵守约束并调用工具。
