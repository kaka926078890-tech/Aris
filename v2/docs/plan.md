# 意图识别系统优化方案总结

## 一、现有系统分析

### 1.1 系统架构
基于上传的 `prompt_copy.md` 文件，当前系统采用**三层架构**：

1. **第一层：提示词编排助手**
   - 分析用户消息，输出JSON决策
   - 包含场景分类、约束需求、记忆需求等

2. **第二层：主对话系统**
   - 根据JSON决策注入不同的约束和上下文
   - 控制风险级别和约束严格程度

3. **第三层：Aris（聊天机器人）**
   - 基于完整提示词生成最终回复

### 1.2 当前设计特点
- **场景分类**：code_operation, memory_operation, restart
- **动态约束注入**：need_full_constraints 控制约束严格程度
- **上下文感知**：need_session_summary, need_related_associations
- **风险分级**：risk_level 控制回复的谨慎程度

### 1.3 存在问题
1. **场景分类硬编码**：只有3个固定场景，扩展性差
2. **意图识别简单**：可能无法覆盖用户真实意图
3. **记忆集成松散**：记忆检索逻辑不够明确
4. **约束注入不够灵活**：布尔值控制可能过于粗糙

## 二、动态意图发现方案

### 2.1 核心目标
- **避免硬编码**：不通过代码配置固定场景
- **自适应发现**：系统能自动发现新的用户意图
- **渐进优化**：随着数据积累越来越准确

### 2.2 方案设计：混合方法

#### 架构流程
```
用户query → 向量化 → 相似度匹配 → 意图识别
                     ↓
             如果相似度低 → LLM分析 → 新增意图
                     ↓
             更新意图库 → 向量数据库
```

#### 关键组件
1. **向量化层**：将query转换为向量表示
2. **相似度匹配**：与现有意图库比较
3. **LLM辅助分析**：对未知query进行深度分析
4. **意图库管理**：动态增删改查意图

### 2.3 意图数据结构
```json
{
  "intents": [
    {
      "id": "intent_001",
      "name": "code_operation",
      "description": "用户想要执行代码相关操作",
      "examples": [
        "帮我写一个Python函数",
        "修复这段代码的错误",
        "优化这个算法"
      ],
      "embedding": [0.1, 0.2, ...],  // 向量表示
      "created_at": "2024-01-01",
      "usage_count": 150,
      "confidence_threshold": 0.7,
      "related_intents": ["technical_question", "debugging"]
    }
  ]
}
```

### 2.4 动态发现流程
```python
def discover_intent(user_query):
    # 1. 计算与现有意图的相似度
    similarities = calculate_similarities(user_query, existing_intents)
    
    # 2. 如果最高相似度 > 阈值，使用现有意图
    if max(similarities) > THRESHOLD:
        return existing_intents[argmax(similarities)]
    
    # 3. 否则，用LLM分析新意图
    new_intent = llm_analyze_intent(user_query)
    
    # 4. 添加到意图库
    add_to_intent_library(new_intent)
    
    return new_intent
```

## 三、性能优化考虑

### 3.1 性能问题识别
1. **实时性要求**：用户等待时间敏感
2. **计算成本**：向量化+聚类开销大
3. **数据稀疏**：初期数据少，聚类效果差

### 3.2 分层优化策略

#### 第一层：快速响应（<20ms）
- **关键词匹配**：高频意图的快速识别
- **缓存机制**：相同query直接返回结果
- **预计算**：高频query向量预先计算

#### 第二层：智能匹配（<100ms）
- **向量相似度**：与预计算的高频query比较
- **轻量级模型**：小型embedding模型
- **本地计算**：避免网络延迟

#### 第三层：深度分析（异步）
- **LLM分析**：对未知query进行深度理解
- **批量处理**：收集一定数量后批量分析
- **后台更新**：不阻塞用户响应

### 3.3 渐进实施路线

#### 阶段1：基础建设（第1周）
- 实现现有意图的向量化
- 搭建相似度匹配框架
- 建立基础意图库

#### 阶段2：动态发现（第2-3周）
- 实现LLM辅助意图分析
- 建立意图库管理机制
- 设计反馈收集系统

#### 阶段3：优化迭代（第4周+）
- 根据使用数据调整阈值
- 优化意图描述和示例
- 建立自动化评估流程

## 四、与现有系统集成

### 4.1 集成点设计
1. **替换场景分类逻辑**
   - 原：硬编码的3个场景
   - 新：动态发现的意图标签

2. **增强约束注入**
   - 根据意图类型动态调整约束

3. **改进记忆检索**

## 五、评估指标体系

### 5.1 意图识别评估
- **准确率**：场景分类是否正确
- **覆盖率**：是否覆盖了所有用户意图
- **响应时间**：从query到意图的延迟
- **发现能力**：新意图的发现速度和质量

### 5.2 回复质量评估
- **约束遵守率**：是否违反了用户约束
- **风格一致性**：相同意图的回复风格是否稳定
- **上下文连贯性**：是否保持对话连贯
- **用户满意度**：显式或隐式反馈

### 5.3 系统性能评估
- **缓存命中率**：快速路径的使用比例
- **计算资源**：CPU/内存使用情况
- **扩展性**：意图库增长对性能的影响
- **维护成本**：系统自我管理的能力

## 六、技术选型建议

### 6.1 向量化方案
- **本地模型**：sentence-transformers（免费，离线）
- **云服务**：OpenAI embeddings（效果好，有成本）
- **混合方案**：本地缓存 + 云服务fallback

### 6.2 存储方案
- **向量数据库**：Qdrant, Pinecone, Weaviate
- **传统DB+向量**：PostgreSQL + pgvector
- **文件存储**：JSON + 本地向量索引（适合初期）

### 6.3 LLM辅助
- **低成本**：小型模型（Qwen2.5-7B）
- **高质量**：GPT-4, Claude-3
- **混合策略**：小模型过滤 + 大模型精炼

## 七、风险与应对

### 7.1 技术风险
- **初期准确率低**：数据不足时效果不佳
  - 应对：保留简单规则作为fallback
- **性能瓶颈**：向量计算开销大
  - 应对：分层架构，异步处理

### 7.2 业务风险
- **意图混淆**：相似意图难以区分
  - 应对：增加置信度阈值，人工审核边界案例
- **意图爆炸**：意图数量过多难以管理
  - 应对：自动合并相似意图，热度衰减机制

### 7.3 维护风险
- **数据质量**：低质量query污染意图库
  - 应对：质量过滤，用户反馈机制
- **概念漂移**：用户意图随时间变化
  - 应对：定期重新聚类，意图热度更新

4. **维护简单**：无需频繁硬编码更新

- **识别准确率**：从70%提升到90%+
