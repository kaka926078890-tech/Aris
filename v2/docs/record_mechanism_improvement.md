# 记录机制改进方案

## 概述

本文档详细分析了当前Aris记录机制存在的问题，并提出了一套完整的改进方案。目标是解决信息覆盖、缺乏关联、依赖低效向量搜索等核心问题，实现日记式累积和智能关联的记忆系统。

## 当前问题分析

### 1. 身份记录覆盖式问题
- **问题**：`writeIdentity`函数在更新名字时会重置整个记录，而不是累积
- **影响**：用户身份信息无法像日记一样自然累积，历史信息丢失
- **示例**：用户多次告知不同身份信息时，只有最后一次被保留

### 2. 要求记录简单追加问题
- **问题**：`appendRequirement`只是简单追加字符串，没有去重、合并或关联逻辑
- **影响**：重复要求堆积，缺乏语义理解，无法建立要求之间的关联
- **示例**：用户多次表达"减少比喻"的要求，会生成多条重复记录

### 3. 缺乏关联性问题
- **问题**：身份、要求、对话记忆、情感记录都存储在独立文件中，没有建立关联
- **影响**：无法形成完整的用户认知图，回忆时缺乏上下文
- **示例**：无法将用户的游戏偏好与其性格特点关联起来

### 4. 依赖低效向量搜索
- **问题**：系统设计矛盾 - 有专门的身份文件存储数据，但实际使用时却依赖向量搜索
- **影响**：结构化信息被向量化后可能丢失，检索效率低下
- **示例**：用户姓名等结构化信息应该直接读取，而不是通过向量搜索

### 5. 系统架构不一致
- **问题**：部分信息使用JSON文件存储，部分使用向量数据库，缺乏统一管理
- **影响**：维护复杂，数据一致性难以保证

## 文件存储结构现状

```
memory/
├── identity.json          # 用户身份（姓名、备注）- 覆盖式
├── requirements.json      # 用户要求列表 - 简单追加
├── corrections.json       # 纠正记录
├── emotions.json         # 情感记录
├── expression_desires.json # 主动表达欲望
lancedb/                  # 向量数据库（对话记忆）
aris_state.json           # 状态信息
aris_proactive_state.json # 主动消息状态
```

## 改进方案

### 短期改进（立即实施）

#### 1. 修改身份记录逻辑
- **目标**：实现累积式更新，支持历史版本
- **实现**：
  - 修改`packages/store/identity.js`中的`writeIdentity`函数
  - 将当前覆盖式更新改为增量式更新
  - 添加时间戳，支持历史回溯
- **数据结构**：
  ```json
  {
    "name": "当前姓名",
    "notes": "当前备注",
    "history": [
      {
        "timestamp": "2024-01-01T10:00:00Z",
        "name": "旧姓名",
        "notes": "旧备注",
        "source": "用户告知"
      }
    ]
  }
  ```

#### 2. 增强要求记录
- **目标**：实现智能去重和语义合并
- **实现**：
  - 修改`packages/store/requirements.js`中的`appendRequirement`函数
  - 添加语义相似度检测（使用嵌入向量）
  - 支持要求合并和关联标签
- **数据结构**：
  ```json
  {
    "requirements": [
      {
        "id": "req_001",
        "text": "减少比喻句",
        "category": "表达方式",
        "tags": ["语言风格", "简洁"],
        "created_at": "2024-01-01T10:00:00Z",
        "updated_at": "2024-01-02T11:00:00Z",
        "frequency": 3,
        "related_to": ["req_002"]
      }
    ]
  }
  ```

#### 3. 创建关联索引
- **目标**：建立信息之间的关联关系
- **实现**：
  - 创建`memory/associations.json`文件
  - 建立身份、要求、对话、情感之间的关联
  - 支持基于主题的检索
- **数据结构**：
  ```json
  {
    "associations": [
      {
        "source_type": "identity",
        "source_id": "name",
        "target_type": "requirement",
        "target_id": "req_001",
        "relationship": "prefers",
        "strength": 0.8
      }
    ]
  }
  ```

### 中期改进（需要重构）

#### 1. 结构化存储设计
- **目标**：创建统一的结构化存储格式
- **实现**：
  - 设计新的`memory/knowledge_base.json`文件
  - 按主题分类存储所有信息
  - 支持嵌套结构和复杂关系
- **核心概念**：
  - **实体**：用户、游戏、话题等
  - **属性**：实体的特征
  - **关系**：实体之间的联系
  - **事件**：时间点上的发生

#### 2. 智能合并系统
- **目标**：新信息与旧信息相关时自动合并
- **实现**：
  - 基于语义相似度的自动合并算法
  - 冲突检测和解决机制
  - 置信度评分系统

#### 3. 时间轴视图
- **目标**：保留信息的演变过程
- **实现**：
  - 所有更改都记录时间戳
  - 支持历史版本对比
  - 可视化时间线

### 长期愿景

#### 1. 日记式累积系统
- 身份信息像日记一样自然累积
- 支持渐进式了解用户
- 保留所有历史痕迹

#### 2. 上下文关联网络
- 所有信息都能建立上下文关联
- 形成完整的认知图
- 支持复杂查询和推理

#### 3. 自我优化机制
- 系统能够自动优化存储和检索
- 学习用户的查询模式
- 自适应调整存储策略

## 实施路线图

### 阶段一：基础改进（1-2周）
1. 修改身份记录逻辑（累积式更新）
2. 增强要求记录（智能去重）
3. 创建基础关联索引

### 阶段二：结构化重构（2-4周）
1. 设计新的结构化存储格式
2. 迁移现有数据到新格式
3. 实现智能合并系统

### 阶段三：高级功能（4-8周）
1. 实现时间轴视图
2. 构建上下文关联网络
3. 添加自我优化机制

## 中长期可执行计划（无硬编码，schema 驱动）

以下为中期、长期的可落地步骤，所有数据结构由 schema 定义，代码不写死字段名。

### 中期（结构化存储 + 智能合并）

| 步骤 | 内容 | 产出/改动 |
|------|------|-----------|
| M1 | 新增 `knowledge_base.schema.json` | 定义 `entities`（实体表）、`relations`（关系表）、`events`（事件表）的字段名与类型；实体含 id、type、attributes（由 schema 子结构描述）、created_at、updated_at。 |
| M2 | 新增 `packages/store/knowledge_base.js` | 实现 `readEntity(id)`、`writeEntity(id, payload)`、`listEntities(type?)`、`addRelation(fromId, toId, relationType, payload)`、`getRelations(fromId?)`；所有键名从 schema 读取。 |
| M3 | 在 `memory_files.json` 中增加 `knowledge_base` 文件名 | 如 `knowledge_base.json` 或按类型分文件（由 schema 或配置决定）。 |
| M4 | 迁移脚本 `scripts/migrate-to-knowledge-base.js` | 读取现有 identity、requirements、associations，按 schema 转成实体与关系并写入 knowledge_base；可配置是否保留原文件。 |
| M5 | 冲突与合并配置 schema | 新增 `merge_policy.schema.json`（或并入 knowledge_base.schema）：`similarity_threshold`、`conflict_strategy`（time_first / confidence_first / ask_user）、可选 `confidence_field`。 |
| M6 | 在 requirements/identity 层或 knowledge_base 层实现合并 | 写新信息时按 merge_policy 查相似实体、解析冲突、更新或新增；调用处仅调统一 API，不硬编码策略。 |

### 长期（时间轴 + 关联网络 + 自优化）

| 步骤 | 内容 | 产出/改动 |
|------|------|-----------|
| L1 | 时间轴 schema 与存储 | 在 knowledge_base.schema 或单独 `timeline.schema.json` 中定义 `history` 结构：每条含 version、timestamp、snapshot 或 diff、actor（user/system）。 |
| L2 | 所有 write 路径写入历史 | identity、requirements、knowledge_base 的每次写操作在写入当前值的同时，向 timeline 表/文件追加一条记录；读取“某时刻状态”时按 timeline 回溯。 |
| L3 | 关联网络查询 API | 在 knowledge_base 或单独模块实现 `getSubgraph(centerId, depth)`、`getPaths(fromId, toId)`，返回的节点与边结构由 schema 定义。 |
| L4 | 检索分层与配置 | 新增 `retrieval.schema.json`：`order`（如 ["structured", "vector"]）、`cache_ttl`、`preload_entities`（类型或 id 列表）；检索层先查结构化再查向量，按配置执行。 |
| L5 | 使用统计与自适应 | 记录各实体/关系的被访问次数与最近访问时间（结构由 schema 定义）；后台或按需任务根据统计调整 merge 阈值、预加载列表、缓存策略。 |

### 原则

- 所有新增/修改的数据结构（字段名、表名、文件名）均在 schema 或 `memory_files.json` 等配置中定义，业务代码只通过 schemaLoader 与路径配置访问。
- 中期先落地 M1–M4，再视需要做 M5–M6；长期在中期稳定后按 L1–L5 分步迭代。

## 技术细节

### 1. 语义相似度检测
- 使用嵌入向量计算文本相似度
- 设置相似度阈值（如0.85）
- 支持多语言和领域特定

### 2. 冲突解决策略
- **时间优先**：最新信息覆盖旧信息
- **置信度优先**：高置信度信息优先
- **用户确认**：重要冲突请求用户确认

### 3. 检索优化
- **分层检索**：先查结构化数据，再查向量库
- **缓存机制**：高频查询结果缓存
- **预加载**：用户常用信息预加载

## 预期效果

### 1. 用户体验提升
- 信息不再被意外覆盖
- 系统更了解用户的历史和偏好
- 交互更加自然和个性化

### 2. 系统性能提升
- 减少不必要的向量搜索
- 提高检索速度和准确性
- 降低内存和存储开销

### 3. 开发维护简化
- 统一的数据管理接口
- 清晰的架构设计
- 易于扩展和调试

## 风险评估与缓解

### 1. 数据迁移风险
- **风险**：迁移过程中数据丢失或损坏
- **缓解**：完整备份，逐步迁移，回滚机制

### 2. 性能影响风险
- **风险**：新机制可能影响响应速度
- **缓解**：性能测试，渐进式部署，监控系统

### 3. 兼容性风险
- **风险**：新格式与现有工具不兼容
- **缓解**：保持向后兼容，提供转换工具

## 总结

本改进方案旨在解决当前记录机制的核心问题，通过渐进式改进实现从简单存储到智能关联的转变。方案注重实用性和可实施性，优先解决最紧迫的问题，同时为长期发展奠定基础。

实施本方案后，Aris将能够：
1. 真正实现日记式累积了解用户
2. 建立完整的用户认知图
3. 提供更加个性化和智能的交互体验
4. 为未来的自我优化和迭代奠定基础
