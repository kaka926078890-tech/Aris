# 体验反馈与优化方向（架构与产品设计）

> 记录当前使用中的四类体验问题，并从 **Agent 架构、模型行为、产品取舍** 角度给出可落地的优化方向。  
> 与实现细节无关的「感受」也如实收录，便于后续版本对照。

---

## 一、反馈原文摘要

| # | 现象 | 用户侧感受 |
|---|------|------------|
| **1** | 工具循环重复度高：读工程时反复 `list` / `read_file`，一层层往下探，即使有目录缓存、文件读缓存，改善有限。 | **工具利用率低**，像在「盲探目录树」。 |
| **2** | 陪伴型需要大量记录感受、体验类内容；回复时走「调用存储 / record / 检索」的流程长，**整体反应慢**，不如普通对话 Bot 流畅。 | **延迟体感差**，与「情绪价值」场景冲突。 |
| **3** | 输出不稳定：风格忽变，时好时坏。 | **一致性不足**，削弱人设信任感。 |
| **4** | 记忆检索、利用记录 / 情绪 / 要求等数据时 **效率低、耗时长**；不确定是提示词还是 API；对比公司内部 Claw Agent，**没有「特别久、流程特别久」** 的感觉。 | **检索与数据链路**成为瓶颈感知。 |

---

## 二、专业判断：根因分层

### 2.1 问题 1 — 工具重复与「递归探路」

**本质**：这是 **LLM 规划失败 + 工具接口粒度** 的叠加，缓存只能减轻 I/O，**不能替代「一次想清楚要看哪」的规划能力**。

- **模型侧**：代码探索是典型的 **long-horizon** 任务；小步 `list → read → list` 是模型默认策略，**除非**在 system 里极强约束「先缓存再列目录、单轮最多 N 次探索」或 **代码侧硬限制**。
- **缓存侧**：当前 `action_cache`（`get_dir_cache` / `get_read_file_cache`）依赖模型 **主动先调缓存工具**；若模型跳过，仍会直接 `list_my_files` / `read_file`，体感上「缓存没用」。
- **接口侧**：若缺少「一次性给项目拓扑 / 关键入口文件」类信息，模型只能 **局部贪心** 往下钻。

**结论**：改善需要 **「行为约束 + 可选更强工具（批量/语义）」**，单靠缓存提示词 **上限明显**。

---

### 2.2 问题 2 — 陪伴型 × 慢：记录与对话抢同一条链路

**本质**：**产品目标（多写入、多检索）** 与 **交互目标（低延迟）** 在同一条 **同步对话管线** 里竞争。

- 每次 `record_*`、`search_memories`、多轮 `chatWithTools` 都是 **额外 RTT**（往返 API）；陪伴场景又容易触发 **多工具串行**。
- **Prompt Planner**（若开启）本身已是 **主对话前多一次完整 LLM 调用**，对「秒回」敏感场景不友好。
- **流式**：主路径上工具轮 **非流式**（`chatWithTools`），用户长时间只看到「在思考」而无 token，**体感延迟**会被放大。

**结论**：需要从 **「哪些写操作异步化」**、**「是否合并 Planner」**、**「首包策略」** 三方面拆，而不是只加工具。

---

### 2.3 问题 3 — 风格不稳定

**本质**：多因素耦合 — **temperature**、**每轮注入块变化大**（Planner 开关、brief/全文、场景块、关联实体行）、**工具结果噪声**、**长 system** 导致模型注意力漂移。

- **非确定性采样** + **上下文变长** → 同一人设下输出方差仍大。
- **情绪与任务**切换时，若缺少「短约束锚点」（如固定收尾句式上限、语气档位），模型会自由发挥。

**结论**：偏 **提示词工程 + 解码参数 + 减少每轮上下文抖动**；必要时 **输出侧轻量规范化**（非删人格，而是限格式）。

---

### 2.4 问题 4 — 检索慢：提示词 vs API？

**可拆解为**：

| 来源 | 说明 |
|------|------|
| **API 延迟** | DeepSeek 每轮工具调用一次完整请求；多轮工具 = 多倍延迟；**向量嵌入走 Ollama 本机**，通常不是主因，除非 Ollama 未起或机器负载高。 |
| **产品设计** | `search_memories` 前可能有 **query 改写 / 加权 / 实体过滤**（见 `memory.js`），逻辑不重，但 **多轮工具** 会累加。 |
| **提示词** | 若模型 **不必要地频繁** `search_memories` 或与 `record` 穿插，流程会变长；属于 **策略问题**，不是单次 API 特别慢。 |
| **与 Claw 对比** | 公司内部 Agent 往往 **不同基础设施**（专有网关、批处理、可能更激进的并行/缓存、或更少工具轮）；**不宜简单归因「我们提示词差」**，更可能是 **架构与调用次数差异**。 |

**结论**：优先 **量化**（一轮对话几次工具、Planner 是否开启、Ollama 是否就绪），再决定是减轮次还是优化检索实现。

## 三、优化方向（按优先级与类型）

### 3.1 架构层

| 方向 | 说明 |
|------|------|
| **异步写入** | 非安全关键的 `record_emotion` / 部分 preference 可 **先回复、后写入**（队列），降低用户感知延迟；需定义 **失败补偿**（重试、落盘）。 |
| **Planner 可选降级** | 对「纯闲聊 / 明显无代码记忆场景」跳过或 **规则粗判**（极轻量），减少固定 **+1 次** DeepSeek；或合并进主模型首轮（单请求输出 plan+json，复杂度高）。 |
| **工具预算** | 每轮或每会话 **最大工具次数 / 最大 explore 深度** 硬限制，超限则返回「请指定文件/路径」—— **反递归探路**。 |
| **批量/高阶文件工具** | 提供 `glob` / `search_in_repo` / 「按关键词列候选文件」等，**减少 list 链**（MVP 可用 ripgrep 封装）。 |

### 3.2 工具与缓存层

| 方向 | 说明 |
|------|------|
| **缓存默认优先** | `read_file` / `list` 在服务端 **自动** 查 `action_cache`，命中则直接返回摘要并在 tool 结果中标注 `from_cache`，**降低对模型自觉性的依赖**。 |
| **探索策略写进场景规则** | 在 `conversation_rules` / 代码场景块中明确：**同一目的连续 list 不得超过 N 次；必须先描述假设路径再读**。 |
| **记忆检索** | 混合检索（稀疏 + 向量）与 **减少候选再喂模型**（见 `memory_update_plan` 思路，Node 侧落地）；避免一次 `search_memories` 拉回过多条再让模型精读。 |

### 3.3 提示词与解码

| 方向 | 说明 |
|------|------|
| **稳定性** | 主对话 `temperature` 略降；Planner 已较低；对「短答」场景可加 **固定输出结构**（非八股，而是「1～3 句 + 可选延伸」）。 |
| **减少 system 抖动** | 稳定 **brief 重建频率**；避免同一会话内「全文 ↔ 摘要」频繁切换。 |
| **工具使用纪律** | 明确：**何时禁止** `search_memories`（极短闲聊）；**何时必须**（用户问「上次说过」）。 |

### 3.4 观测与对标

| 方向 | 说明 |
|------|------|
| **结构化日志** | 每轮记录：`planner_ms`、`tool_rounds`、`tool_names` 序列、`embed_ms`（若有）。便于区分「慢在 API」还是「慢在轮次多」。 |
| **Claw 对标** | 弄清对方：**是否无前置 Planner**、**工具是否并行**、**是否在边缘节点**；再设 **合理目标**（例如 P95 首包 &lt; X s），避免不可比焦虑。 |

## 四、分阶段建议（MVP → 最终）

### MVP（优先落地、改动面可控）

1. **服务端自动查缓存**再读盘（文件工具内短路），减少「模型忘了先 get_*_cache」。  
2. **工具轮上限 + 代码探索深度上限**，防止无限递归。  
3. **会话级策略**：纯闲聊路径 **关 Planner 或** 用环境变量默认关（由你权衡 token vs 延迟）。  
4. **日志**：打每轮工具次数与耗时，验证「慢」的主因。

### 中期

1. **批量/搜索类工具** 减少 list 链。  
2. **异步 record**（情绪类）与 **用户可见「已记下」** 的轻提示。  
3. **记忆检索**：混合检索或更强 query 构造（在现有 LanceDB + Ollama 上迭代）。

### 最终（与产品形态强相关）

1. **单请求规划**（主模型一次输出内部 plan + 回复草稿）— 工程难度大，省延迟最明显。  
2. **独立轻量路由模型**（本地）只做场景分类，替代 Planner 的云端调用 — 成本与维护换延迟。  
3. **检索服务化**（若数据量极大）：与桌面端解耦，当前单用户本地场景 **未必需要**。

## 五、与产品定位的一句话对齐

- **陪伴 / 情绪价值** 依赖 **多记录、多回忆**，天然与 **低延迟** 张力大；优化目标应是 **「关键路径少工具、非关键写入异步、可观测」**，而不是 **零记录**。  
- **工具效率** 的核心是 **减少决策步数**（批量工具、缓存短路、硬限制），不是再加更多说明文字。  
- **风格稳定** 是 **解码 + 上下文稳定 + 输出约束** 的综合题，需单独一轮迭代与 A/B。
