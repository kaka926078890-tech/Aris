# 第二优先级解决方案（ARIS_IDEAS 第二优先级）

针对 ARIS_IDEAS.md 中「第二优先级（近期目标）」的三项，给出可落地的 MVP 与最终方案，供选型与实现参考。

---

## 一、更自然的对话能力

**目标**：更好理解用户情绪与意图（不只是字面意思）；在游戏、累、想安静等情境下自然回应；减少机械感，增加流畅度。

### 现状

- 已有：persona（INFP 朋友、禁止客服用语）、`get_preferences`（喜好/习惯按需拉取）、`record_friend_context`（记录当前情绪/场景）、`last_tired_or_quiet_at` 与「歇会」静默、低功耗模式。
- 缺口：没有显式引导「结合情绪/情境调整语气」；游戏兴趣等虽有偏好记录，但对话中缺少「情境化」提示（如正在聊游戏时多用游戏相关记忆）；回复风格偏统一，情境区分不足。

### MVP 方案

1. **在 rules 中增加「情境与语气」的简短规则（纯 prompt）**
   - 1～2 句：根据当前对话主题与用户情绪调整语气（如聊游戏时可更轻松、用户说累时更简短克制）；避免脱离情境的长篇或固定套路。
   - 不新增配置与工具，只改 rules.md 或 DEFAULT_PERSONA 中的规则说明。

2. **可选：在 buildPromptContext 中注入「当前情境」一句**
   - 若 `recent_mood_or_scene` 或最近一条用户消息包含明确情绪/场景关键词（累、游戏、开心等），在 system 中增加一句极短提示，如「用户当前可能处于：休息/想安静」或「当前话题偏游戏/日常」，不展开长文。
   - 实现：从 state 读 `recent_mood_or_scene`，或对 recent 最后一条 user 做简单关键词匹配，生成一句 10 字以内的情境标签注入；严格限制长度。

3. **游戏兴趣的按需使用**
   - 保持现状：游戏兴趣存 preferences，通过 `get_preferences` 按需拉取。在 persona 或 rules 中加一句：「聊到游戏时，可先调用 get_preferences 查看用户喜欢的游戏，再自然融入对话。」不把整份偏好列表注入每轮。

### 最终方案

1. **可配置的「情境敏感」开关**
   - 在 memory 或 config 中增加如 `context_aware_tone: true`；为 true 时，在 buildPromptContext 中注入上述「当前情境」短句；为 false 时不注入，保持最小 prompt。

2. **轻量情境标签（代码侧）**
   - 在 handler 或单独小模块中，根据最近用户消息与 `recent_mood_or_scene` 生成枚举型标签（如 `tired` / `gaming` / `casual` / `unknown`），仅用于拼一句注入或给后续扩展用，不写入长期存储。

3. **禁止用语与风格示例（规则 + 配置）**
   - 继续在 rules 中用 1～2 句 + 少量示例约束「不说文绉绉、不机械」；长列表放配置文件（如 `avoid_phrases.json`）供人工维护，不在每轮灌入整份列表。

4. **文档**
   - README 或 docs 中说明「情境与语气」相关配置项及可选值；若新增 avoid_phrases 等，按 config-documentation 规则同步文档。

## 二、记忆的连贯性

**目标**：让记忆更像人的记忆——能自然联系过去对话与经历，形成更完整理解，而非碎片化；像日记一样累积而非覆盖。

### 现状

- 已有：identity 累积历史、preferences 按 topic 累积、向量库存储对话与经历、`search_memories` / `search_memories_with_time` / `get_conversation_near_time`、`related_associations` 与近期小结注入 prompt。
- 缺口：身份信息虽有 history，但「当前理解」仍偏单点；检索到的记忆以列表形式呈现，缺少「与当前话题的连贯叙述」；没有「跨会话主题线」的轻量抽象（如「我们常聊的几件事」）。

### MVP 方案

1. **强化「先检索再回复」的提示（纯 prompt）**
   - 在 persona 或 rules 中增加 1 句：需要联系过去经历或用户说过的话时，先调用 search_memories / get_preferences，再基于检索结果组织回复，使对话前后连贯。
   - 不改变检索逻辑，只提升模型对现有工具的使用意愿。

2. **近期小结的更好利用**
   - 已有 `recent_summary` 注入；确保 summaries 模块在适当时机（如会话轮次达到阈值或新会话开始时）写入/更新摘要，内容为「近期几轮或跨会话的简短梗概」，便于模型形成连贯感。若当前实现已满足，仅做文档说明即可。

3. **身份历史的暴露方式**
   - identity 已有 history；在 `get_my_context` 或 read_file 可读的说明中注明「用户身份有历史变更记录，可通过 read_file 查看 identity 文件（若格式允许）」或为 identity 提供「获取最近 N 次变更」的只读接口，供模型按需拉取，不每轮注入整份历史。

### 最终方案

1. **「主题线」或「用户画像」轻量摘要（可选）**
   - 新增一份可由脚本或后台任务生成的轻量文档（如 `memory/user_profile_summary.md`），内容为：常聊主题、最近几次重要偏好变更、近期情绪/场景的简要归纳等，每 N 天或每 M 次对话更新一次；模型通过 read_file 或工具按需获取，不每轮注入。
   - 实现可先采用「人工维护短文」或「从 preferences + 最近对话标题/摘要拼接」的简单版本。

2. **检索结果的呈现优化**
   - 在返回给模型的 `search_memories` 结果中，除列表外增加可选「简短连贯叙述」字段（如 1～2 句）：由代码根据 top-k 结果拼成一句概括，或调用 LLM 做极短摘要（需控制成本与延迟）；便于模型直接引用而非自己拼装。

3. **记忆写入的日记式策略**
   - 确保新增记忆（如 record_preference、对话写入向量）为追加而非覆盖；identity 的 history 已为追加。若有「全局用户状态」类存储，也采用按时间戳追加的条目，便于形成时间线。

4. **文档**
   - 在 README 或 docs 中说明「记忆连贯性」相关设计：身份历史、检索工具、近期小结与（若有）user_profile_summary 的用途与更新策略。

## 三、自我反思和改进

**目标**：能自己发现问题并主动调整；意识到说话太文绉绉或不足时能反思并改进；根据用户反馈调整行为。

### 现状

- 已有：`get_corrections` 可获取用户纠错记录；persona 中已有「减少文绉绉」等要求；rules 可配置。
- 缺口：没有「反思」的固定时机或输出出口；纠错记录被拉取后未强制影响后续风格；缺少「最近用户反馈」的轻量汇总（如最近 N 条 correction 的摘要）。

### MVP 方案

1. **在 rules 中明确「根据纠错调整」**
   - 增加 1 句：若用户曾通过纠错指出用词或风格问题，回复时请参考 get_corrections 的结果，避免重复同类问题。
   - 不新增工具，仅提示模型在需要时调用现有 get_corrections。

2. **可选：首条消息或会话开始时拉取纠错摘要**
   - 在 buildPromptContext 中，若本会话为「首条用户消息」，可注入一句：「用户曾做过的风格/用词纠错（最多 1 句摘要）：xxx」。摘要由代码从 get_corrections 结果取最近 1～3 条拼成一句，严格限制长度（如 30 字内）；若无可不注入。
   - 实现：在 handler 中调用 store 或 tools 获取 corrections，拼成一句写入 CONTEXT_TEMPLATE 的新占位符或追加到 behavioral_rules 末尾。

3. **反思的「出口」最小化**
   - 不要求模型输出结构化「反思」字段；若希望 Aris 有「自我笔记」，可在最终方案中增加「写入自我笔记」工具，MVP 阶段仅依赖 prompt 引导模型在内心调整行为。

### 最终方案

1. **「最近用户反馈」的轻量摘要**
   - 在 memory 或 state 中维护「最近 N 条 corrections 的摘要」或「用户最近指出的问题类型」，在 buildPromptContext 中注入 1 句（可配置开关），便于模型每轮感知到「用户最近在纠正什么」。

2. **自我笔记工具（可选）**
   - 新增工具如 `append_self_note`：Aris 可写入一条仅自己可见的笔记（如「本轮我说话又偏长了，下次更简短」），存储到 memory/self_notes.json 或类似；后续会话可通过 read_file 或专用工具按需读取，形成简单反思链。需在 persona 中说明用途与边界（不写敏感信息、不替代用户配置）。

3. **风格参数的显式化（可选）**
   - 若存在「文绉绉程度」「回复长度倾向」等可配置项，可放在 memory 的配置中，由 Aris 通过工具读取并在回复时自我约束；或由用户通过设置页调整。MVP 可不实现，仅预留设计。

4. **文档**
   - README 或 docs 中说明「自我反思与纠错」：get_corrections 的用途、是否注入摘要、self_notes（若有）的用途与存储位置。

## 四、实施顺序建议

| 顺序 | 项               | 建议                                                                 |
|------|------------------|----------------------------------------------------------------------|
| 1    | 更自然的对话能力   | 先做 MVP：rules 中情境/语气 + 可选「当前情境」一句注入；再视需要加情境标签与配置。 |
| 2    | 记忆的连贯性     | 先做 MVP：强化「先检索再回复」提示、用好近期小结与身份历史；再考虑主题线摘要与检索叙述优化。 |
| 3    | 自我反思和改进   | 先做 MVP：rules 中「参考 get_corrections」+ 可选首条注入纠错摘要；再考虑 self_notes 与反馈摘要。 |

三项均可与第一优先级并行；第二优先级内部可先做「更自然的对话能力」与「自我反思」的 prompt 部分（无代码依赖），再补记忆连贯性的检索与摘要优化。
