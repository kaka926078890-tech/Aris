# 情感与表达记录 — 现状与可选方案

当前做法依赖模型在回复中输出固定格式（【情感摘要】/【表达欲望】），再由正则解析写入向量库。若你觉得这种方式不稳定或不够好，可考虑下面几种替代思路。

---

## 一、当前方式简述

- **写入**：主回复末尾由模型输出「【情感摘要】…」「【表达欲望】…」，handler 用正则解析后写入 LanceDB（`aris_emotion`、`aris_expression_desire`）。
- **使用**：主动消息时按类型检索情感/表达欲望，做优先级与时效分析，决定是否直接输出或再走一轮 LLM。
- **问题**：格式依赖强、解析脆弱；模型可能漏写、乱写或格式不一致，导致记录质量不稳定。

---

## 二、可选方案（仅方案说明，不直接改代码）

### 方案 A：工具化记录（推荐）

- **思路**：不在自然语言里嵌格式，改为模型通过**工具调用**上报。
- **实现要点**：增加工具如 `record_emotion({ description, intensity, tags })`、`record_expression_desire({ text, intensity })`，在主对话的工具列表中提供；模型在需要时调用，主进程收到后直接按结构化写入 LanceDB，不再从回复正文用正则解析。
- **优点**：结构清晰、易校验、不依赖回复文本格式；可与现有 proactive 逻辑兼容（仍按 type 检索）。
- **代价**：要改 prompt 说明「用工具记录情感/表达欲望」、并实现工具与写入逻辑。

---

### 方案 B：事后轻量分析（不依赖模型自觉）

- **思路**：不在主回复里要求模型写【情感摘要】/【表达欲望】，而是**在得到回复后**，用一段短 prompt 或小模型对「本轮回复内容」做一次分析，抽取情绪标签、强度、是否有「想说的话」。
- **实现要点**：handler 在拿到 `reply` 后，再发一次非流式请求，例如「请从下面这段话中提取：1) 情绪标签（如好奇/平静/担忧等）2) 强度 1–5 3) 是否有一句想主动对用户说的话，若有则写出」；或使用小模型/规则（关键词+强度启发式）从回复文本里抽。
- **优点**：主回复不再被格式绑架，记录逻辑集中在一处，便于调参或换模型。
- **代价**：多一次 API 调用或要维护一套抽取规则/小模型。

---

### 方案 C：只保留「上次状态」、不做独立情感库

- **思路**：不再维护 `aris_emotion` / `aris_expression_desire` 向量表；仅保留现有的 `aris_state.json` 里「最近心理状态」等字段。
- **实现要点**：主动消息时只用「上次状态」+ 最近对话 + 时间感，由 LLM 根据这段上下文直接生成「是否想说话 + 内容」，不再检索情感/表达欲望类型。
- **优点**：实现最简单，无格式解析、无多表同步；适合「轻量、少状态」的用法。
- **代价**：没有「积累多轮情感再释放」的精细控制，主动表达更依赖单轮上下文。

---

### 方案 D：按事件触发记录（降低频率）

- **思路**：不每轮都记情感/表达欲望，只在「认为有必要」时记。
- **实现要点**：例如仅当检测到用户消息或回复中有强情绪词、或本轮有纠错、或距离上次记录已过 N 轮时，再触发一次记录（可用工具调用或事后分析）。
- **优点**：减少噪音与写入量，记录更「有事件感」。
- **代价**：需要定义并维护「何时记录」的规则或分类逻辑。

---

### 方案 E：可配置开关

- **思路**：在设置或环境里增加「是否启用情感/表达欲望记录」开关；关闭时不再在 prompt 里要求【情感摘要】/【表达欲望】，也不解析、不写入对应类型。
- **实现要点**：配置项控制 prompt 片段与 handler 内解析/写入分支；proactive 侧在关闭时只走「无情感库」的简化逻辑（如方案 C）。
- **优点**：用户可权衡效果与成本，便于 A/B 或逐步迁移。
- **代价**：多一条配置与分支，需在文档里说明差异。

---

## 三、建议组合

- **短期**：若当前主要痛点是「格式不稳、解析易挂」，可优先考虑 **方案 A（工具化）**，保留现有 proactive 检索逻辑，只把「写入来源」从「正文解析」改为「工具入参」。
- **若希望少改主对话**：可先上 **方案 E（开关）**，关闭时退化为 **方案 C**，再视需要决定是否加 **方案 B** 或 **方案 A**。

以上均为方案级说明；若你选定某一项，再按该方案做具体设计与实现即可。
